	<article id="post-618" class="post-618 post type-post status-publish format-standard hentry category-http category-soap category-upload tag-http tag-soap tag-upload">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/http-upload-vs-soap-request-how-they-are-different/" rel="bookmark">HTTP Upload vs. SOAP Request (how they are different)</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/http-upload-vs-soap-request-how-they-are-different/" title="5:25 pm" rel="bookmark"><time class="entry-date" datetime="2015-05-07T17:25:48+00:00">May 7, 2015</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>What is the difference between an HTTP Upload and a SOAP Request?</p>
<p>An upload is an HTTP request that uses the multipart/form-data content-type. (All HTTP requests and responses are MIME, and the format of a given request is determined by the content-type.) A SOAP request, on the other hand, has a content-type of text/xml, which means it is composed of a MIME header + body, where the body contains the SOAP XML. Whereas an upload is a multipart MIME structure, where each sub-part contains either the content of a file, or a name/value parameter.</p>
<p>Here are some examples. First a very simple HTTP upload as produced by this C++ fragment:</p>
<blockquote>
<pre>CkUpload upload;
CkUrl url;

url.ParseUrl("https://www.some-web-server.com/receiveTheUpload.aspx");

upload.put_Hostname(url.host());
upload.put_Path(url.path());
upload.put_Port(url.get_Port());

upload.AddCustomHeader("MyCustomRequestHeader","abc 123");

upload.AddFileReference("TinyA","qa_data/xml/tinyA.xml");

CkByteData uploadData;
upload.UploadToMemory(uploadData);
uploadData.saveFile("qa_output/uploadData1.txt");
</pre>
</blockquote>
<p>The HTTP request (which is MIME, and by the way, and emails are also composed of MIME, so understanding a bit about MIME will do one good because it is so central to Internet technologies..) looks like this:</p>
<blockquote>
<pre>POST /receiveTheUpload.aspx HTTP/1.1
Host: www.some-web-server.com
User-Agent: ChilkatUpload/3.0 (http://www.chilkatsoft.com/)
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8
Connection: keep-alive
Expect: 100-continue
MyCustomRequestHeader: abc 123
Content-Type: multipart/form-data; boundary=---------------------------DD1A174A4121EDFD
Content-Length: 206

-----------------------------DD1A174A4121EDFD
Content-Disposition: form-data; name="TinyA"; filename="tinyA.xml"
Content-Type: text/xml

&lt;tiny&gt;A&lt;/tiny&gt;
-----------------------------DD1A174A4121EDFD--
</pre>
</blockquote>
<p>A SOAP XML request, as sent by this fragment of C++ code:</p>
<blockquote>
<pre>CkHttp http;

http.put_SessionLogFilename("qa_output/xmlPostLog.txt");

// The URL used here is junk and will result in a 404 not found. 
// (We don't care about the response, we
// are only interested in understanding the format of the HTTP request sent.)
// Imagine the XML passed in the 2nd argument is actually a SOAP XML document... 

CkHttpResponse *resp = http.PostXml("http://www.chilkatsoft.com/mywebservice",
    "&lt;tiny&gt;A&lt;/tiny&gt;","utf-8");
delete resp;
</pre>
</blockquote>
<p>Looks like this:</p>
<p>(Imagine the XML is actually a SOAP XML document..)</p>
<blockquote>
<pre>POST /mywebservice HTTP/1.1
Content-Type: text/xml
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8
Connection: keep-alive
User-Agent: Mozilla/5.0 (Windows NT 6.3; WOW64; rv:28.0) Gecko/20100101 Firefox/28.0
Accept-Language: en-us,en;q=0.5
Accept-Encoding: gzip
Host: www.chilkatsoft.com
Content-Length: 14

&lt;tiny&gt;A&lt;/tiny&gt;
</pre>
</blockquote>
<p>An HTTP upload containing two file is produced by this fragment of code:</p>
<blockquote>
<pre>CkUpload upload;
CkUrl url;

url.ParseUrl("https://www.some-web-server.com/receiveTheUpload.aspx");

upload.put_Hostname(url.host());
upload.put_Path(url.path());
upload.put_Port(url.get_Port());

upload.AddCustomHeader("MyCustomRequestHeader","abc 123");

upload.AddFileReference("TinyA","qa_data/xml/tinyA.xml");
upload.AddFileReference("TinyB","qa_data/xml/tinyB.xml");

CkByteData uploadData;
upload.UploadToMemory(uploadData);
uploadData.saveFile("qa_output/uploadData2.txt");
</pre>
</blockquote>
<p>And the HTTP request looks like this:</p>
<blockquote>
<pre>POST /receiveTheUpload.aspx HTTP/1.1
Host: www.some-web-server.com
User-Agent: ChilkatUpload/3.0 (http://www.chilkatsoft.com/)
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8
Connection: keep-alive
Expect: 100-continue
MyCustomRequestHeader: abc 123
Content-Type: multipart/form-data; boundary=---------------------------6F87F47E0974A38B
Content-Length: 363

-----------------------------6F87F47E0974A38B
Content-Disposition: form-data; name="TinyA"; filename="tinyA.xml"
Content-Type: text/xml

&lt;tiny&gt;A&lt;/tiny&gt;
-----------------------------6F87F47E0974A38B
Content-Disposition: form-data; name="TinyB"; filename="tinyB.xml"
Content-Type: text/xml

&lt;tiny&gt;B&lt;/tiny&gt;
-----------------------------6F87F47E0974A38B--
</pre>
</blockquote>
<p>Finally, an HTTP upload with multiple files and a few name/value params is produced by this C++ fragment:</p>
<blockquote>
<pre>CkUpload upload;

CkUrl url;

url.ParseUrl("https://www.some-web-server.com/receiveTheUpload.aspx");

upload.put_Hostname(url.host());
upload.put_Path(url.path());
upload.put_Port(url.get_Port());

upload.AddCustomHeader("MyCustomRequestHeader","abc 123");

upload.AddParam("ParamA","This is the value for param A");
upload.AddParam("ParamB","This is the value for param B");

upload.AddFileReference("TinyA","qa_data/xml/tinyA.xml");
upload.AddFileReference("TinyB","qa_data/xml/tinyB.xml");

CkByteData uploadData;
upload.UploadToMemory(uploadData);
uploadData.saveFile("qa_output/uploadData3.txt");
</pre>
</blockquote>
<p>And the HTTP request looks like this:</p>
<blockquote>
<pre>POST /receiveTheUpload.aspx HTTP/1.1
Host: www.some-web-server.com
User-Agent: ChilkatUpload/3.0 (http://www.chilkatsoft.com/)
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8
Connection: keep-alive
Expect: 100-continue
MyCustomRequestHeader: abc 123
Content-Type: multipart/form-data; boundary=---------------------------EFC8C527CC836355
Content-Length: 617

-----------------------------EFC8C527CC836355
Content-Disposition: form-data; name="ParamA"

This is the value for param A
-----------------------------EFC8C527CC836355
Content-Disposition: form-data; name="ParamB"

This is the value for param B
-----------------------------EFC8C527CC836355
Content-Disposition: form-data; name="TinyA"; filename="tinyA.xml"
Content-Type: text/xml

&lt;tiny&gt;A&lt;/tiny&gt;
-----------------------------EFC8C527CC836355
Content-Disposition: form-data; name="TinyB"; filename="tinyB.xml"
Content-Type: text/xml

&lt;tiny&gt;B&lt;/tiny&gt;

</pre>
</blockquote>

	<article id="post-579" class="post-579 post type-post status-publish format-standard hentry category-http category-release-notes tag-http tag-release-notes tag-resumedownload">

						<h1 class="entry-title"><a href="http://cknotes.com/v9-5-0-35-micro-update-http-resume-download-fixed-percentdone-event-callback/" rel="bookmark">v9.5.0.35 Micro Update: HTTP Resume Download &#8211; Fixed PercentDone Event Callback</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/v9-5-0-35-micro-update-http-resume-download-fixed-percentdone-event-callback/" title="8:13 am" rel="bookmark"><time class="entry-date" datetime="2014-04-25T08:13:13+00:00">April 25, 2014</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->

				<div class="entry-content">
			<p>The PercentDone callback always began at 0 when resuming a download. If a partial file exists locally, and the download is resumed, the percent done now begins at the place based on the size of the partial file. (If 50% of the file already exists, then the percent done begins at 50%)</p>
<p>Also, the Socket.BuildHttpGetRequest method was re-introduced. This is a convenience method for building a simple HTTP GET request. For any more complex HTTP GET requests, use the Chilkat.HttpRequest object instead. (This method was removed temporarily due to cross-linkage with all of the internal HTTP infrastructure which caused additional unused baggage to be linked into an executable. It was rewritten to avoid this linkage.)</p>

	<article id="post-578" class="post-578 post type-post status-publish format-standard hentry category-http category-release-notes category-tls tag-asn-1 tag-http tag-release-notes tag-ssl tag-tls">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/v9-5-0-34-micro-update-ssltls-perfect-forward-secrecy-minor-http-and-asn-1-fixes/" rel="bookmark">v9.5.0.34 Micro Update: SSL/TLS Perfect Forward Secrecy, Minor HTTP and ASN.1 Fixes</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/v9-5-0-34-micro-update-ssltls-perfect-forward-secrecy-minor-http-and-asn-1-fixes/" title="8:57 am" rel="bookmark"><time class="entry-date" datetime="2014-04-24T08:57:12+00:00">April 24, 2014</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>The internal Chilkat SSL/TLS implementation now supports the TLS_DHE_RSA_WITH_AES_256_CBC_SHA and TLS_DHE_RSA_WITH_AES_128_CBC_SHA cipher suites. These allow for perfect forward secrecy. Note: This is implemented on the client-side for all protocols using SSL/TLS. The SSL/TLS client provides the server with a list of algorithms it supports, and it is the server that decides which is to be used. These new algorithms are now included in the list, and will be used if the server chooses.</p>
<p>Also, a minor problem was fixed in Chilkat HTTP. If a server responds with no Content-Length header, then there was a chance that Chilkat would not return the full response. This does not apply to &#8220;chunked&#8221; responses &#8212; only to non-chunked responses that are lacking the Content-Length header &#8212; which is a rare occurrence (and it is poor practice for an HTTP response to omit the Content-Length).</p>
<p>Finally, a minor and rarely encountered ASN.1 problem was fixed. (ASN.1 code is internal to Chilkat and has to do with implementations for PKCS, PFX, etc.</p>

	<article id="post-576" class="post-576 post type-post status-publish format-standard hentry category-http category-release-notes tag-abort tag-event tag-http tag-release-notes">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/v9-5-0-32-micro-update-http-connection-left-in-invalid-state-after-event-callback-aborts-operation/" rel="bookmark">v9.5.0.32 Micro Update: HTTP Connection Left in Invalid State after Event Callback Aborts Operation</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/v9-5-0-32-micro-update-http-connection-left-in-invalid-state-after-event-callback-aborts-operation/" title="12:39 pm" rel="bookmark"><time class="entry-date" datetime="2014-04-19T12:39:33+00:00">April 19, 2014</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>For HTTP requests that have KeepAlive indicated in the response header, the internal connection is not closed so that a subsequent request may continue using the existing connection. However, when the client application aborts in the middle of receiving the response, the connection must be closed to avoid being left in an invalid state (where the some of the remainder of the aborted request may have already been received and would incorrectly be read as the first bytes of the nexxt response). The internal fix was to always close the connection after an abort so that the next HTTP operation starts afresh on a new connection.</p>
					</div><!-- .entry-content -->

	<article id="post-425" class="post-425 post type-post status-publish format-standard hentry category-http tag-http tag-synchronousrequest">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/understanding-url-syntax-and-arguments-to-httpsynchronousrequest/" rel="bookmark">Understanding URL Syntax (and Arguments to Http.SynchronousRequest)</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/understanding-url-syntax-and-arguments-to-httpsynchronousrequest/" title="9:30 am" rel="bookmark"><time class="entry-date" datetime="2012-07-06T09:30:54+00:00">July 6, 2012</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>URL&#8217;s have this general format:</p>
<blockquote><p><tt>&lt;scheme&gt;://</tt><tt>&lt;</tt><tt>domain</tt><tt>&gt;</tt><tt>:</tt><tt>&lt;</tt><tt>port</tt><tt>&gt;</tt><tt>/</tt><tt>&lt;</tt><tt>path</tt><tt>&gt;</tt><tt>?</tt><tt>&lt;</tt><tt>query_string</tt><tt>&gt;</tt><tt>#</tt><tt>&lt;</tt><tt>fragment_id</tt><tt>&gt;</tt></p></blockquote>
<p>The URL parts are summarized below.  Comments about how each part relates to the SynchronousRequest method are included.  The SynchronousRequest method has the following signature (C# syntax)</p>
<blockquote><p>HttpResponse SynchronousRequest(string domain, int port, bool ssl, HttpRequest req);</p></blockquote>
<p>The URL parts:</p>
<ul>
<li>scheme:  This can be &#8220;http&#8221; or &#8220;https&#8221;.   If &#8220;https&#8221;, then SSL/TLS is used and the &#8220;ssl&#8221; argument to SynchronousRequest should be set to <em>true</em>.</li>
<li>domain:  The domain name or dotted IP address of the web server hosting the resource.  This is the <em>domain </em>argument to SynchronousRequest</li>
<li>port: The port number on which the server is listening.  If not explicitly specified, the default for non-SSL is 80, the default for SSL/TLS is 443.  This value must be explicitly passed in the <em>port</em> argument to SynchronousRequest.</li>
<li>path:  The local name for the resource on the server.  In SynchronousRequest, the path is passed in the <em>Path </em>property of the HttpRequest object (which is the last argument to SynchronousRequest).</li>
<li>query_string:  It may contain name/value pairs separated by ampersands, for example <tt>first_name=John&amp;last_name=Doe</tt>.  If SynchronousRequest is sending a GET request (where the HttpRequest.HttpVerb = &#8220;GET&#8221;, then the HttpRequest.Path property should contain the entire path + query string.</li>
<li>fragment_id:  A name for a piece or part of the resource.  The fragment_id is never passed to the server and is only used by the client (i.e. the browser).  Typically, it&#8217;s to automatically navigate to a specific anchor within the HTML page.</li>
</ul>
