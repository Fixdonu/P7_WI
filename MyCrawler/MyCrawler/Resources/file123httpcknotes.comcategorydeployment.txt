	<article id="post-212" class="post-212 post type-post status-publish format-standard hentry category-deployment category-x64 tag-clickonce tag-x64">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/clickonce-w-x64-dlls/" rel="bookmark">ClickOnce w/ x64 DLL&#8217;s</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/clickonce-w-x64-dlls/" title="5:25 am" rel="bookmark"><time class="entry-date" datetime="2009-09-15T05:25:23+00:00">September 15, 2009</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>I recently discovered this crucial bit of information:</p>
<blockquote><p>I compile my project for x86 systems (vs Any CPU or x64). Just wanted you to be aware of this. Your last email stated &#8220;You must deploy the x64 ChilkatDotNet2.dll to x64 systems&#8221; which is not correct.  This is an important consideration for &#8220;Click Once&#8221; deployments which do not have built-in conditional installations as do MSI installations. So by compiling for x86, &#8220;Click Once&#8221; deployments need only distribute the 32 bit version of Chilkat for both 32 and 64 bit systems.</p></blockquote>
<p>My understanding has been refined to this:  It is possible to run either x86 or x64 EXE&#8217;s on the x64 architecture, however, DLL&#8217;s loaded by the EXE must match the architecture of the EXE (x86 or x64).  My guess is that an EXE created in a managed .NET application compiled to &#8220;Any CPU&#8221; would be required to load a DLL matching the processor architecture.</p>
<p>The Windows Registry can also be an issue.  This post describes how the <a title="x64 registry" href="http://cknotes.com/?p=97">registry on x64 Windows is divided into 32-bit and 64-bit keys</a>.  If your EXE is loading an ActiveX DLL (not a .NET assembly) then an x86 EXE would be reading the x86 keys, but if the ActiveX registry information is in the x64 keys, you&#8217;ll get an error when trying to create the object &#8212; probably a &#8220;class not registered&#8221; type of error.</p>
					</div><!-- .entry-content -->

				<div class="entry-content">
			<p><strong>Question:</strong><br />
Very quick question for you about distribution.  Is it a simple matter of just using this unlock code in the appropriate place in the code and adding a reference to the dll in my project?  Or, do I need to include and run your MSI with the installation of my own application for all of my customers?</p>
<p><strong>Answer:</strong><br />
You don&#8217;t need to run the .msi &#8212; it&#8217;s simply a matter of inserting the permanent unlock code into the UnlockComponent method call.</p>
<p>(If you&#8217;re using the ActiveX, you&#8217;ll also need to make sure the Chilkat DLL is registered via regsvr32 &#8212; and that is standard to all ActiveX&#8217;s and handled automatically by most installer programs.)</p>
