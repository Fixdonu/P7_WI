	<article id="post-400" class="post-400 post type-post status-publish format-standard hentry category-error-messages category-incorrect-format category-x64 tag-net tag-64-bit tag-errors tag-incorrect-format tag-x64">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/net-assembly-incorrect-format-error/" rel="bookmark">.NET Assembly Incorrect Format Error</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/net-assembly-incorrect-format-error/" title="9:13 am" rel="bookmark"><time class="entry-date" datetime="2012-02-07T09:13:44+00:00">February 7, 2012</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<pre>
"Could not load file or assembly 'ChilkatDotNet2, Version=9.3.0.0, Culture=neutral,
PublicKeyToken=eb5fc1fc52ef09bd' or one of its dependencies.
 An attempt was made to load a program with an incorrect format."
</pre>
<p>The &#8220;incorrect format&#8221; error is a 32-bit vs. 64-bit mismatch.  It means the application is trying to load     a 64-bit .NET assembly into a 32-bit process,  or the reverse (a 32-bit assembly into a 64-bit process).</p>
<p>Possible solutions are:</p>
<p>1) Target your app for x86 instead of &#8220;Any CPU&#8221;.  If you do this,     then your application will run as a 32-bit process regardless of the     machine architecture and you may always use the 32-bit Chilkat     assembly.</p>
<p>2) If you really need to target &#8220;Any CPU&#8221;, then make sure to deploy     the 32-bit Chilkat assembly w/ your app on 32-bit systems, and     deploy the 64-bit Chilkat assembly to 64-bit systems.</p>
<p>3) If your app is running within ASP.NET or a Windows Service and     you&#8217;re not quite sure whether it&#8217;s 32-bit or 64-bit, then install     the 32-bit Chilkat assembly in the 32-bit GAC (Global Assembly     Cache), and also install the 64-bit Chilkat assembly into the 64-bit     GAC.  The .NET runtime will automatically look in the correct GAC     and will find the correct assembly.</p>
					</div><!-- .entry-content -->

				<div class="entry-content">
			<p><strong>Link Error:</strong></p>
<blockquote><p>Undefined symbols for architecture i386:</p>
<p>&#8220;operator new(unsigned long)&#8221;, referenced from:<br />
-[CkoMailMan init] in libchilkatIos.a(CkoMailMan.o)<br />
&#8220;operator delete(void*)&#8221;, referenced from:<br />

The Chilkat internals are written in C++. Therefore, your Objective-C application will need to link against the C++ runtime libraries. The easiest way to do it is to include an empty source file with the file extension &#8220;cpp&#8221; in your app target.</p>

	<article id="post-346" class="post-346 post type-post status-publish format-standard hentry category-error-messages category-sftp tag-error-messages tag-permission-denied tag-sftp">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/sftp-permission-denied-w-openfile-method/" rel="bookmark">SFTP Permission Denied w/ OpenFile Method</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/sftp-permission-denied-w-openfile-method/" title="6:27 am" rel="bookmark"><time class="entry-date" datetime="2010-10-25T06:27:48+00:00">October 25, 2010</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p><strong>Question:</strong></p>
<p>We are trying to upload a file to SFTP using private key. We are able to connect and retrieve folder listing. But when we try to upload to one of the folder on the SFTP we get the “Permission denied” error message.</p>
<p>The message is as below.</p>
<pre>ChilkatLog:
  OpenFile:
    DllDate: Feb 15 2010
    UnlockPrefix: ****
    Username: joe
    Component: .NET 2.0
    SshVersion: SSH-1.99-OpenSSH_3.8p1
    SftpVersion: 3
    filename: .\Upload\myFiles\test.xml
    access: writeOnly
    createDisposition: createNew
    v3Flags: 0x2a
    Sent FXP_OPEN
    StatusResponse:
      Request: FXP_OPEN
      StatusCode: 3
      StatusMessage: Permission denied
    SshLog:
SFTP&gt; Sending SSH_FXP_OPEN
TRAN&gt; CHANNEL_DATA
TRAN* NumBytes: 44
TRAN&lt; CHANNEL_DATA
SFTP&lt; Received SSH_FXP_STATUS
</pre>
<p><strong>Answer:</strong></p>
<p>The content of the LastErrorText property (as shown above) provides good information that allows me to suggest some things to try:</p>
<ol>
<li>I can see by the &#8220;DllDate&#8221; that the version being used is older.  I would highly recommend updating to the latest version to ensure you&#8217;re getting the best information in LastErrorText.  The information in LastErrorText is always being refined and improvedd as Chilkat does user-support day-to-day.</li>
<li>The createDisposition passed to OpenFile in this case is &#8220;createNew&#8221;.  If the file already exists, then the OpenFile will fail.  It&#8217;s possible that the server will indicate the failure as a &#8220;Permission denied&#8221; error.  Instead, use &#8220;createTruncate&#8221; for the createDisposition argument.</li>
<li>I see that backslashes are used in the remote file path.  Try using forward slashes instead.</li>
<li>To fully understand the absolute file path of the destination file on the SSH/SFTP server, call RealPath to ask the server to convert a relative path to an absolute path.  Maybe the SSH/SFTP user account&#8217;s home directory is something different than expected.</li>
</ol>

						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/vb6-error-%e2%80%9ctype-mismatch-error-code-13%e2%80%9d-w-imap-search-method/" title="8:16 am" rel="bookmark"><time class="entry-date" datetime="2010-10-05T08:16:53+00:00">October 5, 2010</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->

				<div class="entry-content">
			<p>This happens if the Chilkat ActiveX objects are dragged and dropped onto your formvia the “Project–>Components” menu.  Instead, add a reference to the objects via the “Project–>References” menu.   </p>
<p>Instances of the object should be created dynamically.  For example:</p>
<pre>
Dim imap as ChilkatImap
Set imap = New ChilkatImap
</pre>

						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/vc-link-problem-fatal-error-c1047-chilkatlib-was-created-with-an-older-compiler-than-other-objects-rebuild-old-objects-and-libraries/" title="5:05 am" rel="bookmark"><time class="entry-date" datetime="2010-08-25T05:05:47+00:00">August 25, 2010</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->

				<div class="entry-content">
			<p>Regarding the following VC++ link problem:</p>
<pre>
“fatal error C1047: The object or library file 'c:\Chilkat\libs\x64\ChilkatRel_x64.lib' 
was created with an older compiler than other objects; rebuild old objects and libraries”
</pre>
<p>This can be resolved by building without the /LTCG flag turned on. Unfortunately, it is not possible to use the /LTCG flag if you are using the Chilkat libraries. Compiling and linking your C++ application without using the /LTCG flag should resolve the problem.</p>

		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/imap-login-bad-command-received-in-invalid-state-2/" rel="bookmark">IMAP Login: BAD Command received in Invalid state.</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/imap-login-bad-command-received-in-invalid-state-2/" title="5:23 am" rel="bookmark"><time class="entry-date" datetime="2010-08-19T05:23:23+00:00">August 19, 2010</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>This error can happen if you connect to an IMAP server without SSL/TLS (i.e. at port 143) but the server does not allow unsecure sessions.  Oddly enough, the Microsoft Exchange IMAP4 server will accept the connection, but will fail any login attempt with this error.  Here is a sample SessionLog:</p>
<pre>
Connecting to IMAP server at xxxx.com:143
> ----IMAP RESPONSE----
> * OK The Microsoft Exchange IMAP4 service is ready.
> ----IMAP REQUEST----
> aaab LOGIN "mylogin"
> ----IMAP RESPONSE----
> aaab BAD Command received in Invalid state.
</pre>
<p>The solution is to use port 993 w/ SSL/TLS.  Do this by setting the Chilkat.Imap.Port property = 993, and the Chilkat.Imap.Ssl property = True prior to connecting.</p>

	<article id="post-327" class="post-327 post type-post status-publish format-standard hentry category-activex category-error-messages category-regsvr32 tag-301 tag-regsvr32">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/error-0x8002801d-when-instantiating-chilkat-http-activex/" rel="bookmark">Error 0x8002801D when Instantiating Chilkat HTTP ActiveX</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/error-0x8002801d-when-instantiating-chilkat-http-activex/" title="5:04 am" rel="bookmark"><time class="entry-date" datetime="2010-08-19T05:04:16+00:00">August 19, 2010</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>This error indicates that there is a<br />
problem with the registry information for the ActiveX DLL. The registry<br />
entry may be missing or contain incorrect information, or the user may<br />
not have permission to read the registry entry.</p>
<p>In ASP, the error message may look like this:   <strong><br />
Server object error &#8216;ASP 0177 : 8002801d&#8217;</strong></p>
<p>In applications, you may see an error message such as this:  <strong><br />
OLE error code 0x8002801d: Library not registered</strong></p>
<p>First try unregistering and re-registering the DLL.  On the computer or server where the ChilkatHttp.dll is located, open a DOS prompt and &#8220;cd&#8221; to the directory where the DLL is located.  Then unregister by typing:</p>
<blockquote><p>regsvr32 -u ChilkatHttp.dll</p></blockquote>
<p>Next, register the ActiveX DLL by typing:</p>
<blockquote><p>regsvr32 ChilkatHttp.dll</p></blockquote>
<p>If it fails, then create a .bat file in the same directory as the DLL.  Type the &#8220;regsvr32 ChilkatHttp.dll&#8221; command into the .bat script.  Right-click on the .bat and select &#8220;Run as Administrator&#8221;.  This should register the DLL.   Check to see if the problem is resolved in your ASP page or application.</p>
<p>Finally, if the problem remains, it could be a permissions problem.  (I have no idea why the permissions<br />
might be different..)  In any case, please see the information at this Microsoft<br />
support knowledge-base article.  It provides details about changing the<br />
registry entry permissions so that your ASP code has permission to read<br />
the registry entries:</p>
<p>http://support.microsoft.com/kb/274038</p>
