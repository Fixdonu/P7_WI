						<h1 class="entry-title"><a href="http://cknotes.com/java-to-copy-byte-tofrom-a-ckbytearray/" rel="bookmark">Java to copy byte[] to/from a CkByteArray</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/java-to-copy-byte-tofrom-a-ckbytearray/" title="8:14 am" rel="bookmark"><time class="entry-date" datetime="2017-07-19T08:14:49+00:00">July 19, 2017</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>The Chilkat online reference documentation for the CkByteData class, concerning the methods for copying Java byte arrays into and out of a CkByteData object, are incorrect.</p>
<p>This is the correct way to do it:</p>
<pre class="brush: java; title: ; notranslate" title="">
CkByteData byteData = new CkByteData();

// Create a Java byte array with some bytes..
byte[] myData = &quot;Any String you want&quot;.getBytes();

// Copy the bytes from the Java byte array into the CkByteData.
byteData.appendByteArray(myData);

// Copy the bytes from the CkByteData to a new Java byte array.
byte[] myData2 = byteData.toByteArray();
</pre>

		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/using-the-chilkat-activex-in-vb6/" rel="bookmark">Using the Chilkat ActiveX in VB6</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/using-the-chilkat-activex-in-vb6/" title="2:26 pm" rel="bookmark"><time class="entry-date" datetime="2015-09-23T14:26:28+00:00">September 23, 2015</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>To use the Chilkat ActiveX in a Visual Basic 6.0 program, review each of these items to make sure you&#8217;re doing everything correctly:</p>
<ol>
<li>Make sure you&#8217;re using the 32-bit ActiveX, even on 64-bit Windows.</li>
<li>The ActiveX must be registered (via regsvr32) on each computer.  If you&#8217;re having trouble, try using the .msi installer available here: <a href="http://cknotes.com/chilkat-activex-msi-installers/">Chilkat 32-bit ActiveX MSI Installer</a></li>
<li>The ActiveX should be added to the VB6 project by adding a reference.  Add a reference from the VB6 menu “Project–&gt;References”.  Do NOT &#8220;add the component&#8221;.  Do NOT add the ActiveX via the “Project–&gt;Components…” menu item.  You should NOT see anything added to the VB6 palette.  The Chilkat objects are non-visual and should always be instantiated dynamically using the New keyword.  Do NOT drag-and-drop a Chilkat object onto a form.  (This should not be possible if you&#8217;ve only added a reference.)</li>
<li>If the first method call on an object crashes, check to see if the object was created.  Check to see if the variable is Nothing prior to calling the method.  If there is still a problem, check to see if you can examine the Version property.  All Chilkat objects have a Version property.  If the access to the Version property crashes, the ActiveX object was never instantiated.  It may not be registered correctly, or your program may have never created an instance via the New keyword.</li>
<li>If you see a .oca file, this is not something provided by Chilkat.  It is something VB6 generated.  If you recently updated to a new version of the Chilkat ActiveX, delete the old .oca.</li>
</ol>

	<article id="post-630" class="post-630 post type-post status-publish format-standard hentry category-uncategorized">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/chilkat-binary-encoding-list/" rel="bookmark">Chilkat Binary Encoding List</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/chilkat-binary-encoding-list/" title="5:26 am" rel="bookmark"><time class="entry-date" datetime="2015-06-20T05:26:39+00:00">June 20, 2015</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>Throughout the Chilkat API there are method arguments and properties to indicate an encoding. These are encodings for binary data such as &#8220;hex&#8221;, &#8220;base64&#8221;, etc.</p>
<p>Note: These are not character encodings. A character encoding, such as &#8220;utf-8&#8221; indicates the byte representation of a character. A binary encoding, such as &#8220;base64&#8221;, represents non-character binary data in printable string form.</p>
<p>Wherever an <em>encoding</em> method argument or property exists, the list of valid encodings supported by Chilkat are as follows:</p>
<ul>
<li>base64</li>
<li>hex</li>
<li>modbase64</li>
<li>base64url</li>
<li>base32</li>
<li>base58</li>
<li>uu</li>
<li>qp (for quoted-printable)</li>
<li>url (for url-encoding)</li>
<li>Q (for MIME Q-Encoding)</li>
<li>B (for MIME B-Encoding)</li>
<li>url_oauth</li>
<li>url_rfc1738</li>
<li>url_rfc2396</li>
<li>url_rfc3986</li>
<li>fingerprint (added in Chilkat v9.5.0.55)</li>
<li>decimal (added in Chilkat v9.5.0.55)</li>
<li>eda (added in Chilkat v9.5.0.65, for encoding/decoding to the UN/EDIFACT Syntax Level A character set)</li>
<li>json (added in Chilkat v9.5.0.66, for escaping and unescaping JSON strings)</li>
<li>decList (added in Chilkat v9.5.0.66. See the notes below.)</li>
<li>base64_mime (added in Chilkat v9.5.0.67. Same as &#8220;base64&#8221; but the output is in lines just as it would appear in MIME or in a PEM.)</li>
</ul>
<p>Chilkat is case-insensitive for recognizing an encoding by name.  For example &#8220;Base64&#8221; and &#8220;base64&#8221; are both recognized as &#8220;base64&#8221;.</p>
<p>The &#8220;fingerprint&#8221; encoding is a lowercase hex encoding where each hex digit is separated by a colon character.  For example: 6a:de:e0:af:56:f8:0c:04:11:5b:ef:4d:49:ad:09:23</p>
<p>The &#8220;decimal&#8221; encoding is for converting large decimal integers to/from a big-endian binary representation.  For example, the decimal string &#8220;72623859790382856&#8221; converts to the bytes 0x01 0x02 0x03 0x04 0x05 0x06 0x07 0x08.</p>
<p>The &#8220;decList&#8221; encoding is for converting comma-separated lists of decimal integers to bytes and back.  For example, &#8220;84, 104, 101, 32, 116, 114, 117, 101, 32, 115, 105, 103, 110&#8221;.  Each decimal integer is a value from 0 to 255.</p>

	<article id="post-552" class="post-552 post type-post status-publish format-standard hentry category-release-notes category-uncategorized tag-release-notes">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/chilkat-v9-5-0-release-notes/" rel="bookmark">Chilkat v9.5.0 Release Notes</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/chilkat-v9-5-0-release-notes/" title="8:32 pm" rel="bookmark"><time class="entry-date" datetime="2014-04-14T20:32:25+00:00">April 14, 2014</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>* As with any new release, some fixes occur deep within the internal Chilkat code base and are applicable to many classes/objects.  These are labeled as &#8220;Internal&#8221; fixes.</p>
<p><strong>FTP2: </strong>Fixed XCRC problem.</p>
<p><strong>HTTP: </strong>Added  the <em>AwsEndpoint</em> property.  This is the regional endpoint (domain) to be used for Amazon S3 method calls.  The default value is &#8220;s3.amazonaws.com&#8221;.  This can be set to any valid Amazon S3 endpoint, such as &#8220;s3-eu-west-1.amazonaws.com&#8221;, or the endpoints for S3-API compatible services from other different providers.</p>
<p><strong>HTTP/Upload: </strong>Added the Upload.<em>Expect100Continue</em> boolean property.  Also modified the HTTP class so that if the &#8220;Expect: 100-continue&#8221; header is included, then the intermediate HTTP response is first received before continuing with sending the data of the HTTP request.</p>
<p><strong>Zip: </strong>Fixed a ZipEntry problem.  If an existing .zip is opened, and an entry&#8217;s <em>Filename</em> property is modified, then the re-written .zip did not have the updated filename.  Also fixed a ZipEntry.<em>ReplaceString</em> problem.</p>
<p><strong>TAR: </strong>Fixed long path problem with the ustar format.</p>
<p><strong>MHT: </strong>The Mht.UseCids property will now default to False because using Content-Location seems to be more robust than using Content-ID for embedded images. (IE showed broken images even though the Content-ID and &#8220;CID&#8221; URLs were correct.)</p>
<p><strong>UnixCompress: </strong>Fixed an uncompress problem.</p>
<p><strong>Zip: </strong>The <em>Encryption</em> property did not initialize when a .zip is opened. (Technically, a zip archive may have entries with different properties.  Each entry within a zip is independent of the others.  There is nothing to prevent some entries from being encrypted, while other entries are unencrypted.  The typical behavior is that all entries are encrypted or unencrypted, but this is purely as a result of the common behavior found in zip utility programs.  Therefore, the Zip.Encryption property is set to the value of the 1st non-directory entry found within a .zip that is opened.</p>
<p><strong>IMAP: </strong>In some rare cases, when downloading an email excluding attachments, such that the email has a zero-length body, then the email download hangs (until a timeout) and the download fails.</p>
<p><strong>Android/iOS C/C++:  </strong>Added the &#8220;C&#8221; functions and includes to the iOS and Android C/C++ libs.</p>
<p><strong>IMAP: </strong>Fixed problem with <em>FetchBundle</em> when downloading emails and excluding attachments.</p>
<p><strong>SSH Tunnel: </strong>When a CHANNEL_CLOSE is received from the SSH Server, the tunnel is now closed.</p>
<p><strong>FTP2: </strong>Directory listings missing user/group information are now correctly handled.</p>
<p><strong>SFTP: </strong>Fixed rare problem with <em>DownloadFileByName</em> that caused a hang.</p>
<p><strong>HTTP: </strong>Params added by the HttpRequest.<em>AddParam</em> method were not added to the &#8220;start line&#8221; for text/xml or other types of requests.</p>
<p><strong>Objective-C: </strong>PercentDone events for MailMan, Zip and Ftp2 fixed.</p>
<p><strong>Internal: </strong>For domain lookups, only IPv4 was used.  Now both IPv4 and IPv6 is possible, with preference automatically given to IPv4.  A new PreferIpv6 property will become available at some point in the future to allow preference to be given to IPv6 if both types of addresses are available.</p>
<p><strong>SFTP: </strong>Fixed potential multi-threading problem with SFTP downloading of files.</p>
<p><strong>SFTP</strong>: Improved memory usage for the <em>ReadDir</em> method.</p>
<p><strong>FTP2</strong>: Added the <em>SyncMustMatch</em> and <em>SyncMustNotMatch</em> properties.</p>
<p><strong>Email Object</strong>: Removed the <em>SetFromMimeObject</em> and <em>GetMimeObject</em> methods.</p>
<blockquote><p>This is to eliminate the linkage between the Mime and Email objects which can result in much larger executables for apps linking with C/C++ libs (such as iOS). Existing apps that used these methods should instead accomplish the same in 2 steps: (1) Get the MIME as a string, (2) Load the object from the MIME string.  This is effectively what was happening in the removed methods, so there is no performance loss &#8212; it&#8217;s simply a matter of replacing 1 line of code with 2 lines of code..</p></blockquote>
<p><strong>MHT</strong>: Removed the GetEmail, GetMime, and HtmlToEmail methods because these return objects and cause much larger executables for apps linking with C/C++ libs.  Removing these methods disconnects MHT from Email/MIME objects for smaller executables which is important in some environments.  The same workaround as described above can be used to accomplish the same: (1) call the method that returns the MIME string, such as GetMHT or GetEML, and then (2) load the MIME string into a new instance of an Email or Mime object.</p>
<p><strong>HTTP</strong>: Fixed a case in HttpRequest.AddHeader where an extra SPACE char was getting inserted after a semicolon in some header field values.</p>
<p><strong>Delphi DLL (non-ActiveX)</strong>:  The *.pas sources now include the Chilkat DLL in this way:</p>
<blockquote><p>{$Include chilkatDllPath.inc}</p>
<p>The chilkatDllPath.inc file is a single location where the path to the Chilkat DLL may be modified.  It contains this:</p>
<p>{$IFDEF WIN32}<br />
const DLLName = &#8216;ChilkatDelphiXE.dll&#8217;;<br />
{$ELSE}<br />
{$IFDEF WIN64}<br />
const DLLName = &#8216;ChilkatDelphiXE64.dll&#8217;;<br />
{$ELSE}<br />
const DLLName = &#8216;ChilkatDelphiXE.dll&#8217;;  //Older versions of Delphi<br />
{$ENDIF}<br />
{$ENDIF}</p></blockquote>
<p><strong>Zip</strong>: The <em>OpenFromWeb</em> method was removed.  There are two reasons: (1) The HTTP related functionality creates dependencies on all that is related to HTTP/SSL/encryption/etc. which greatly expands the size of any EXE using zip functionality, and this is undesired on mobile platforms.  (2) The HTTP functionality is oversimplistic, because it lacks many features (such as proxies, and other HTTP complexities) that might be required.  It is better if a program separately uses the Chilkat HTTP functionality to download a .zip (into memory or into a file) and then open it with Chilkat Zip.</p>
<p><strong>Zip</strong>: Removed the Proxy property, which only applied to OpenFromWeb.</p>
<p><strong>Socket</strong>: Removed the BuildHttpGetRequest method to eliminate the Socket&#8211;&gt;Http dependency, to help reduce size of C++ builds.  The HTTP API can instead be used to create HTTP GET requests.</p>
<p><strong>Events</strong>:  C++ event callbacks have been modified to used 64-bit integers where appropriate (CkZipProgress, and others).  When using inheritance for event callbacks, such as w/ C++, check to make sure your derived class matches the method signatures that are now 64-bit.  One way to ensure that your event callback classes correctly match the base class method signatures is to declare methods using the #define&#8217;s found in the Ck*Progress.h headers.  For example:</p>
<blockquote><p>#define CK_FTP2PROGRESS_API \<br />
void BeginDownloadFile(const char *pathUtf8, bool *skip);\<br />
void EndDownloadFile(const char *pathUtf8, __int64 numBytes);\<br />
void VerifyDownloadDir(const char *pathUtf8, bool *skip);\<br />
void BeginUploadFile(const char *pathUtf8, bool *skip);\<br />
void EndUploadFile(const char *pathUtf8, __int64 numBytes);\<br />
void VerifyUploadDir(const char *pathUtf8, bool *skip);\<br />
void VerifyDeleteDir(const char *pathUtf8, bool *skip);\<br />
void VerifyDeleteFile(const char *pathUtf8, bool *skip);\<br />
void UploadRate(__int64 byteCount, unsigned long bytesPerSec);\<br />
void DownloadRate(__int64 byteCount, unsigned long bytesPerSec);</p></blockquote>
<p><strong>FTP2</strong>: Directory listings are now guaranteed to be transferred in binary mode to avoid certain charset/interpretation problems.</p>
<p><strong>Zip</strong>: The AppendAnsi and AppendUnicode are removed because AppendString2 encompasses the same functionality and more.</p>
<p><strong>CkZipProgress</strong>: Added the isDirectory arg to ToBeUnzipped and FileUnzipped callback methods.  This corrected a discrepency between the callback function definitions in different programming languages.</p>
<p><strong>RSA</strong>: For OAEP signatures, fixed a problem in verification for signatures (where the last byte was 0xBC but just by chance..)</p>
<p><strong>HTTP</strong>: HTTP requests are automatically re-tried if the request was sent on an open/persistent connection, but the disconnected state is not found until trying to read the response header.</p>
<p><strong>TAR</strong>: Fixed a problem with writing 8GB or greater size files (not the size of the entire file, but when a single file contained within the TAR is 8GB or greater).</p>
<p><strong>Delphi ActiveX</strong>: All object creation is now uniform and consistent.  This is reflected in the online examples an reference documentation.</p>
<p><strong>Email</strong>: Fixed this problem: If text/csv attachments used non-usascii chars, then the bytes within the attachment were converted to the charset of the email (treating the attachment content like the text/plain or text/html body of the email.</p>
<p><strong>MessageSet</strong>: The ToString method was renamed to ToCommaSeparatedStr because &#8220;ToString&#8221; has conflicts in the .NET environment.</p>
<p><strong>Email Object</strong>: The GetLinkedDomains method was modified to return a CkStringArray object instead of passing by reference.</p>
<p><strong>FTP2</strong>: The UploadRate/DownloadRate properties were renamed to UploadTransferRate/DownloadTransferRate because the names conflicted with events having the same names.</p>
<p><strong>SFTP</strong>: Fixed a ReadDir internal parsing problem when FXP_NAME&#8217;s have extensions (rare).</p>
<p><strong>SFtpFile</strong>: The FileType property never returned symLink, charDevice, blockDevice, fifo, etc.</p>
<p><strong>Single-DLL ActiveX</strong>: Most ActiveX object names ending in &#8220;2&#8221; are changed so that the &#8220;2&#8221; is removed: MailMan, Email, EmailBundle, Zip, ZipEntry, Charset.   The Crypt2 and Ftp2 remain as-is because these agree with the names used in the other programming languages.</p>
<p><strong>IMAP</strong>: Fixed rare IMAP response parsing problem.</p>
<blockquote><p><strong>Single DLL ActiveX</strong> methods removed:<br />
** These are changes in the Single-DLL ActiveX.  The individual (deprecated) ActiveX DLLs are unchanged.<br />
MailMan: VerifyRecipients, NewEmail, NewBundle<br />
ZipEntry: InflateToString, InflateToString2, InflateToString3 (use UnzipToString instead)<br />
Xmp: NewCkStringArray<br />
Xml: CompressNode/DecompressNode (use ZipContent/UnzipContent instead)<br />
Xml: CompressSubtree/DecompressSubtree (use ZipTree/UnzipTree instead)<br />
Xml: DecryptNode/EncryptNode (use DecryptContent/EncryptContent instead)<br />
Xml: LoadXmlV<br />
Upload: VariantToString<br />
StringArray: Intersect (use Subtract instead)<br />
Spider: Sleep (use SleepMs instead), GetDomain (use GetUrlDomain instead)<br />
Bounce: Removed ActiveX&#8217;s ExamineMail (use ExamineEmail instead)<br />
Imap, Http, Ftp2: Removed unused last arg from SetSslClientCertPfx in ActiveX<br />
Cert: IsExpired &#8212; use the Expired property instead.<br />
CertStore: LinkForSigning (no longer needed)<br />
Charset.UrlDecode &#8212; use UrlDecodeStr instead.<br />
Emailbundle.AddMime &#8211;&gt; AddMimeBytes (this is an ActiveX only method..)<br />
Email.AddRelatedData method reverses args to conform with .NET, C++, etc.<br />
Email.LoadTextFileToUnicode removed &#8212; use CkString.LoadFile instead.<br />
KeyContainer.GetContainerName removed &#8212; call GetNthContainerName instead.<br />
HttpRequest.Serialize removed<br />
HttpResponse.Serialize/Unserialize removed<br />
Ftp2: DllInfo/DllPath properties removed.</p></blockquote>
<p><strong>MHT</strong>: Will not modify image URLs with query params if they are not to be embedded.</p>
<p><strong>HTTP</strong>: Various issues with proxy authentication resolved.</p>
<p><strong>IMAP</strong>: Added new methods to the Mailboxes class/object: GetFlags,GetMailboxIndex,GetNthFlag,GetNumFlags,HasFlag</p>
<p><strong>IMAP</strong>: Added the Capability method.</p>
<p><strong>Internal</strong>: For SSL/TLS channel closure, if initiating the shutdown, Chilkat will now wait for the close notify alert response from the server after sending its close notify alert (for a cleaner shutdown)</p>
<p><strong>Internal</strong>: Fixed ipv6 hostname issues w/ HTTP where the IP address was being truncated at the 1st colon character.</p>
<p><strong>DKIM</strong>: Fixed DKIM problem where a null byte was sometimes added to the MIME when AddDkimSignature was called.</p>
<p><strong>Objective-C</strong>: The Objective-C API now has the full set of event callbacks making it identical to the C++ API.</p>
<p><strong>TAR/Gzip</strong>: Fixed problem in UnTarGz where an improperly terminated tar archive (i.e. a corrupted archive) caused in infinite loop within Chilkat.</p>
<p><strong>Zip</strong>: Fixed QuickAppend for when the .zip grows beyond 4GB to automatically use ZIP64 format.</p>
<p><strong>C++</strong>: Added the CkBaseProgress class as the base class for progress events (Ck*Progress).   All future Chilkat C++ classes will (try to) use only the standard CkBaseProgress events rather than having events specific to each class/object.  Therefore, if a new Chilkat class, named &#8220;Abc&#8221; is created, then the corresponding event class will be CkBaseProgress instead of CkAbcProgress.   If for some reason special events are needed, then a CkAbcProgress class can be added that derives from CkBaseProgress.  The event callbacks in the CkBaseProgress could also be extended in the future.</p>
<p><strong>MHT</strong>: The &#8220;DownloadingUrl&#8221; event is removed from CkMhtProgress because it was only present in the C++ API and not in .NET or ActiveX.  It is<br />
now a &#8220;ProgressInfo&#8221; event with the tag &#8220;DownloadingUrl&#8221;.</p>
<p><strong>C++</strong>: Methods that passed binary bytes as a combination of &#8220;const unsigned char *pByteData, unsigned long szByteData&#8221; are changed to conform to the standard of passing a CkByteData reference instead.   (A CkByteData can &#8220;borrow&#8221; data to avoid memory-to-memory copying.)</p>
<blockquote><p>The following methods were modified:<br />
CkEmail::BEncodeBytes<br />
CkEmail::QEncodeBytes<br />
CkEmail::AddDataAttachment<br />
&#8212; for AddDataAttachment, in the ActiveX the order of the args reverses to agree with other languages.<br />
&#8212; The Objective-C arg names change.<br />
CkEmail::GetMbHeaderField2 is removed.<br />
CkEmail::GetMbHeaderField args change to: fieldName, charset, returns bytes.<br />
CkEmail::SetMbHtmlBody<br />
CkEmail::SetMbPlainTextBody<br />
CkCharset::SetErrorBytes<br />
CkSocket::AsyncSendBytes  (Objective-C arg name changes)<br />
CkSocket::SendBytes  (CkSocket::SendByteData removed) (Objective-C arg name changes)<br />
Removed CkZip::AppendData2 (use AppendData instead)<br />
CkZip::OpenFromMemory (Objective-C arg name changes)</p></blockquote>
<p><strong>Internal</strong>: Fixed URL parsing where a URL ending with a port number, such as &#8220;http://192.168.1.105:8888&#8221;, without any final forward slash char, the port number did not get parsed.</p>
<p><strong>CSV</strong>: Fixed issue with backslash chars (double backslash must be used for input, but leading backslash needed to be removed when fetching content)</p>
<p><strong>IMAP</strong>: Fixed problem with downloading emails without attachments.  In some cases, attachments were downloaded that should not have been.</p>
<p><strong>IMAP</strong>: Fixed ListMailboxes for a case where a non-use-ascii string is passed for the wildcardedMailbox.</p>
<p><strong>Internal</strong>: Fixed a rare iso-2022-jp charset conversion problem.</p>
<p><strong>Email Object</strong>: The EmailDate and LocalDate will now both return local times.  The reason is that for anything returning a date/time in SYSTEMTIME, it will always be a local time. However, the SYSTEMTIME methods are deprecated and will not be used in new Chilkat classes or components.  The methods that use CkDateTime objects should be used instead.</p>
<p><strong>Zip</strong>: Fixed problem  when re-writing zip archives having internal data descriptors.</p>
<p><strong>Internal</strong>: Fixed problem with zero-length PFX/P12 passwords.</p>
<p><strong>SSH/SFTP</strong>: IF the SSH password is empty, and keyboard-interactive is allowed by the server, then keyboard-interactive is chosen over password authentication.</p>
<p><strong>MIME/Internal</strong>: Fixed Q/B decoding which could result in extra SPACE chars getting inserted into a MIME header.</p>
<p><strong>MHT</strong>: Added code to recognize jpeg/exif and other types of jpg variants.</p>
<p><strong>Email Object</strong>: SetReplacePatterns did not replace all possible header fields.</p>
<p><strong>FTP2</strong>: The XML returned by DirTreeXml now includes size and last-modified date/time as XML attributes for each file.</p>
<p><strong>Internal (Windows-only)</strong>: SSL/TLS implementation may now be able to use client-side certs having non-exportable private keys.</p>
<p><strong>XML</strong>: Double and single quotes in content are not automatically encoded to &amp;quot; and &amp;apos;</p>
<p><strong>Zip</strong>: Fixed ZIP64 writing problem.</p>
<p><strong>MIME/Internal</strong>: Fixed iso-2022-jp issue that can sometimes occur with MIME header parsing.</p>
<p><strong>DirTree/FileAccess</strong>: Added FileSize64 property to DirTree ActiveX and FileSize64 method to FileAccess ActiveX.</p>
<p><strong>HTTP</strong>: Added the BasicAuth property (must be set to true for the HTTP component to send a request using basic authentication).</p>
<p><strong>HTTP</strong>: Fixed the HttpBeginReceive, HttpEndReceive, HttpBeginSend, HttpEndSend event callbacks.</p>
<p>&nbsp;</p>

						<h1 class="entry-title"><a href="http://cknotes.com/download-sftp-files-indirectly-through-a-web-site/" rel="bookmark">Download SFTP Files Indirectly through a Web Site</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/download-sftp-files-indirectly-through-a-web-site/" title="10:01 am" rel="bookmark"><time class="entry-date" datetime="2010-12-13T10:01:05+00:00">December 13, 2010</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p><strong>Question:</strong></p>
<p>I am doing a project that requires a web site to list files from an SFTP  server and allow the visitor to download them.  Following the examples  on your site, it was very easy to list the files.  However, you have no  example of downloading a file to the user, only to the local filesystem.   Do you have any code that could help me with this?</p>
<p><strong>Answer:</strong></p>
<p>This is a common question.  To restate the situation:</p>
<ol>
<li>There  is an SSH/SFTP server on your private network accessible from the server-side code running on your web server, but not accessible to the outside world.</li>
<li>The need is to deliver files from this server to an end-user located anywhere on the Internet using only a browser.</li>
</ol>
<p>The solution:</p>
<ol>
<li>The end-user clicks on a link or form &#8220;submit&#8221; button that sends an HTTP request (GET or POST) to your web server.</li>
<li>The page handling this request includes server-side code, which (of course) runs on the web server.  It needs to download the file from the SSH/SFTP server to either a temporary file (on the web server) or perhaps directly into an in-memory byte array.</li>
<li>The file data is then sent in an appropriate HTTP response, with correct Content-Type header indicating the type of data.  If using ASP/ASP.NET, it will probably involve calling Response.BinaryWrite.</li>
</ol>

						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/eset-nod32-false-positive-genetik-trojan/" title="6:18 am" rel="bookmark"><time class="entry-date" datetime="2009-06-15T06:18:05+00:00">June 15, 2009</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->

				<div class="entry-content">
			<p>Recently, the ESET NOD32 AntiVirus program began erroneously detecting the Chilkat .NET 2.0/3.5 download as containing a Genetik Trojan virus.  This is incorrect (i.e. it is a false positive).  After researching the Genetik Trojan, it has become apparent that NOD32&#8217;s virus definition for this particular virus is frequently a false-positive.  If you experience this problem, <strong>report the false-positive by sending email to samples@eset.com</strong>.</p>
<p>NOTE: <strong>The problem only occurs during the download.</strong>  The ChilkatDotNet2.dll, once on your system, is not detected as a virus.  Therefore, you may bypass the problem by downloading the Chilkat .NET assembly from this encrypted .zip:  </p>
<p><a href="http://www.chilkatsoft.com/download/ChilkatDotNet2_enc.zip">http://www.chilkatsoft.com/download/ChilkatDotNet2_enc.zip</a><br />
The password is &#8220;chilkatsoft&#8221;.  </p>
<p>Unzip and place the ChilkatDotNet2.dll in your app&#8217;s bin directory. In Visual Studio, add a reference to this assembly by browsing to the DLL&#8217;s location and selecting the DLL.</p>
