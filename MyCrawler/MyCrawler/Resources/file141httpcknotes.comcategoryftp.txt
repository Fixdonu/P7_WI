		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/damn-you-windows-firewall/" rel="bookmark">Damn You Windows Firewall!</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/damn-you-windows-firewall/" title="5:47 am" rel="bookmark"><time class="entry-date" datetime="2016-07-03T05:47:04+00:00">July 3, 2016</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>One particular cause of the following error is finally known:</p>
<pre>sockRecv(47ms):
    WindowsError: An existing connection was forcibly closed by the remote host.
    WindowsErrorCode: 0x2746
    numBytesRequested: 5
    Failed to receive data on the TCP socket
--sockRecv
sockRecv failed.
sockRecvN_buf: Did not receive the exact number of bytes desired.
numBytesToReceive: 5</pre>
<p>This error was caused by the Windows Firewall doing stateful FTP filtering. The solution is to disable stateful FTP filtering so that the firewall does not block any FTP traffic. (This is typically prescribed for the server-side of FTP, but it also applies to the client-side.)</p>
<p><strong>Note:</strong> The Windows Firewall&#8217;s stateful FTP filtering only causes problems when the connection is SSL/TLS. Clear (non-secured) connections are not affected.</p>
<p>To disable stateful FTP filtering, open an <strong>administrative</strong> command prompt, and type the following:</p>
<pre>netsh advfirewall set global StatefulFTP disable</pre>
<p><strong>Also Note:</strong> The above error could also be a result of the connection truly being closed by the remote host, or something on the server-side. This solution only applies to the case where Windows Firewall (on the client-side computer) is interfering with a SSL/TLS FTP connection.</p>
<p><strong>How to Reproduce the Problem</strong></p>
<p>To reproduce the problem, write a program that establishes a TLS connection with an FTP server.   After authenticating, write a simple loop that calls ftp.Noop 1500 times.   If Windows Firewall is interfering, it is likely to fail in the same place every time.  For example, in my case the error occurs on iteration 699 every time.</p>
<p>Also, the error may not be easily reproducible even if Windows Firewall is a problem.  For example, the problem seemed to be not reproducible if the FTP server was not a Windows-hosted FTP server.   One might suspect the server-side to be the problem, but the connection reset ([RST, ACK] in a WireShark trace) definitely originated from the client-side.</p>
<p>Disabling the firewall&#8217;s Stateful Packet Inspection (SPI) feature solved the problem every time. (It should be the case that stateful packet inspection should NEVER try to inspect the packets of a TLS encrypted channel for the simple fact that it&#8217;s impossible to inspect.  The firewall does not have the ability to decrypt the packets in the channel, and therefore it cannot inspect anything. The firewall&#8217;s only choices SHOULD be to simply allow or disallow the connection.  I don&#8217;t understand why stateful packet inspection should ever be &#8220;monkeying around&#8221; with encrypted channels..)</p>

	<article id="post-449" class="post-449 post type-post status-publish format-standard hentry category-ftp tag-ftp-timeout">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/ftp-timeouts-on-large-files/" rel="bookmark">FTP Timeouts on Large Files</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/ftp-timeouts-on-large-files/" title="7:07 am" rel="bookmark"><time class="entry-date" datetime="2013-01-29T07:07:16+00:00">January 29, 2013</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>Here is a description of the problem from the FileZilla documentation at http://wiki.filezilla-project.org/Network_Configuration#Timeouts_on_large_files</p>
<p>A few possible workarounds are described below.</p>
<blockquote><p><strong>Timeouts on large files</strong></p>
<p>If you can transfer small files without any issues, but transfers of larger files end with a timeout, a broken router and/or firewall exists between the client and the server and is causing a problem.</p>
<p>As mentioned above, FTP uses two TCP connections: a control connection to submit commands and receive replies, and a data connection for actual file transfers. It is the nature of FTP that during a transfer the control connection stays completely idle.</p>
<p>The TCP specifications do not set a limit on the amount of time a connection can stay idle. Unless explicitly closed, a connection is assumed to remain alive indefinitely. However, many routers and firewalls automatically close idle connections after a certain period of time. Worse, they often don&#8217;t notify the user, but just silently drop the connection. For FTP, this means that during a long transfer the control connection can get dropped because it is detected as idle, but neither client nor server are notified. So when all data has been transferred, the server assumes the control connection is alive and it sends the transfer confirmation reply. Likewise, the client thinks the control connection is alive and it waits for the reply from the server. But since the control connection got dropped without notification, the reply never arrives and eventually the connection will timeout.</p>
<p>In an attempt to solve this problem, the TCP specifications include a way to send keep-alive packets on otherwise idle TCP connections, to tell all involved parties that the connection is still alive and needed. However, the TCP specifications also make it very clear that these keep-alive packets should not be sent more often than once every two hours. Therefore, with added tolerance for network latency, connections can stay idle for up to 2 hours and 4 minutes.</p>
<p>However, many routers and firewalls drop connections that have been idle for less than 2 hours and 4 minutes. This violates the TCP specifications (RFC 5382 makes this especially clear). In other words, all routers and firewalls that are dropping idle connections too early cannot be used for long FTP transfers. Unfortunately manufacturers of consumer-grade router and firewall vendors do not care about specifications &#8230; all they care about is getting your money (and only deliver barely working lowest quality junk).</p>
<p>To solve this problem, you need to uninstall affected firewalls and replace faulty routers with better-quality ones.
</p></blockquote>
<p><strong>Some Possible Workarounds</strong></p>
<p>1) Set the SoSndBuf and SoRcvBuf properties equal to 131072, which is the default used by FileZilla.  If this improves the transfer rate, then it may be that the transfer takes a shorter amount of time and the timeout of whatever is interfering is not exceeded.</p>
<p>2) Set the SkipFinalReply property = true (or 1 if using the ActiveX).   This will cause the FTP2 object to avoid trying to get the final control-channel reply.  However, you should (A) close the connection and re-connect because if the final control-channel reply did arrive but remained unread, then the FTP client and server will be out-of-sync in terms of commands and replies.  (In other words, the next FTP command sent on the control channel will get the previous reply.)      (B) When using SkipFinalReply, write code to verify the size of the remote file after the transfer.  </p>

	<article id="post-427" class="post-427 post type-post status-publish format-standard hentry category-ftp tag-ftp tag-listpattern">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/listpattern-property-setting-can-affect-how-ftp2-gets-directory-information/" rel="bookmark">ListPattern Property Setting can Affect how FTP2 Gets Directory Information</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/listpattern-property-setting-can-affect-how-ftp2-gets-directory-information/" title="5:48 am" rel="bookmark"><time class="entry-date" datetime="2012-07-17T05:48:08+00:00">July 17, 2012</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>If an FTP server supports the <a href="http://tools.ietf.org/html/rfc3659#page-23">RFC 3659 extension</a>, which includes the MLSD command, then it is automatically known to Chilkat FTP2 because the FTP server&#8217;s response to the FEAT command will include MLSD in the list.  (Chilkat automatically sends a FEAT command after connecting so that it knows the capabilities of the FTP server.)</p>
<p>When Chilkat FTP2 recognizes that MLSD is supported, it will use this extension whenever possible.  The reason is that MLSD returns more accurate date/time and size information.  The drawback is that it can only return a full directory listing.  Therefore, if the ListPattern property is set to anything other than &#8220;*&#8221;, such as &#8220;*.txt&#8221;, then MLSD cannot be used.  </p>
<p>When MLSD is not used, Chilkat must use the FTP protocol&#8217;s LIST command which returns the directory listing in a human-readable format.  The problem with the LIST response is that different operating systems and FTP servers respond with different listing formats.  Chilkat FTP2 automatically recognizes and parses all known directory formats, including formats for uncommon systems such as VAX/VMS, MVS, AS/400, etc.    Another issue with the human-readable response is that date/time information may be inaccurate (for example, timezone may not be known, or for older files the HH:MM:SS may not be provided).  </p>
<p>Therefore, when using FTP2 to examine the contents of a remote directory, if accurate date/time information is important, then make sure the ListPattern is set to &#8220;*&#8221; and write your application code to iterate over all the files in the remote directory, perhaps only acting on the files that have names matching what your app needs.</p>
<p>Of course, even when using &#8220;*&#8221;, if the FTP server itself does not support MLSD, then your app is at the mercy of whatever information is or is not included in the human readable directory listing returned by the LIST command. </p>

	<article id="post-411" class="post-411 post type-post status-publish format-standard hentry category-ftp category-sftp tag-ftp tag-ftps tag-sftp tag-ssh">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/sftp-vs-ftps/" rel="bookmark">SFTP vs. FTPS</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/sftp-vs-ftps/" title="3:32 pm" rel="bookmark"><time class="entry-date" datetime="2012-03-13T15:32:45+00:00">March 13, 2012</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p><strong>Clarification on the acronyms &#8220;SFTP&#8221; and &#8220;FTPS&#8221;</strong></p>
<p>&#8220;SFTP&#8221; is the <strong>S</strong>ecure <strong>F</strong>ile <strong>T</strong>ransfer <strong>P</strong>rotocol over SSH.  It is a protocol unrelated to the FTP protocol. (It is actually a subsystem of SSH.)  The Chilkat SSH / SFTP component is used for &#8220;SFTP&#8221;.  SFTP is achieved by connecting to an SSH server at port 22.</p>
<p>On the other hand, the Chilkat FTP2 component is for FTP.   FTP servers listen at port 21 (non-SSL/TLS) and port 990 (SSL).  FTP over SSL (i.e. port 990) is called &#8220;FTPS&#8221;.</p>

		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/ftps-530-no-client-certificate-presented/" rel="bookmark">(FTPS) 530 No client certificate presented.</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/ftps-530-no-client-certificate-presented/" title="7:43 am" rel="bookmark"><time class="entry-date" datetime="2012-01-24T07:43:43+00:00">January 24, 2012</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>If this FTP server response is seen in the LastErrorText, it means that the SSL/TLS connection requires a client-side certificate with private key.   Prior to connecting, the client-side certificate should be specified by calling the SetSslClientCert, or SetSslClientCertPfx.  Make sure to check the return value of these methods for success/failure.  If the SetSslClientCert* method fails, then no client-side certificate has been set, and the Chilkat SSL/TLS handshake implementation will send a 0-length client-side certificate, which is the correct response for servers that request a client-side cert but don&#8217;t actually require it.</p>
<p>If the certificate object passed to SetSslClientCert is a pre-installed certificate on the Windows operating system, then make sure it was installed from the PFX such that the private key is exportable.  Chilkat&#8217;s SSL/TLS handshake code needs access to the client-side private key when a client-side cert is required.</p>

		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/ftp-directory-synchronization-and-removing-files/" rel="bookmark">FTP Directory Synchronization and Removing Files</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/ftp-directory-synchronization-and-removing-files/" title="7:11 am" rel="bookmark"><time class="entry-date" datetime="2011-08-26T07:11:47+00:00">August 26, 2011</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>Chilkat FTP2 provides methods for synchronizing a local directory tree with a remote directory tree &#8212; in both directions.  The SyncLocalTree method <strong>downloads </strong>files from the remote tree to the local tree, and SyncRemoteTree <strong>uploads </strong>files from the local tree to the remote tree.  Both have a number of different &#8220;modes&#8221; that determine which files are transferred, depending on existence, last-modified date comparison, etc.  However, one thing that is intentionally missing is a feature to automatically delete files that do not exist on the other side.  The synchronization is only additive, not deletive.  There is a workaround though, as explained below.</p>
<p><strong>Case #1: Deleting files in the remote tree that do not exist in the local tree. </strong></p>
<p>Determine which files need to be deleted by calling <strong>SyncLocalTree </strong>using mode     #1.  Write an event callback handler for OnBeginDownloadFile (see     <a class="moz-txt-link-freetext" href="http://www.chilkatsoft.com/p/p_385.asp">http://www.chilkatsoft.com/p/p_385.asp</a> ) to skip every download so     that nothing is acutally downloaded.  Instead, you&#8217;ll build a list     of files that would&#8217;ve been downloaded because they exist on the     server but not locally.  Then write a loop that iterates over the     list and calls DeleteRemoteFile for each.</p>
<p><strong>Case #2: Deleting files in the local tree that do not exist in the remote tree.</strong></p>
<p>The same technique is used.  Determinen which files need to be deleted by calling SyncRemoteTree using mode #1.  Write an event callback handler for OnBeginUploadFile to skip every upload so that nothing is actually uploaded.  Instead, you&#8217;ll build a list of files that would&#8217;ve been uploaded because they exists locally but not on the server.  Then write a loop that iterates over the list can deletes each local file.</p>
					</div><!-- .entry-content -->

						<h1 class="entry-title"><a href="http://cknotes.com/solved-425-unable-to-build-data-connection-operation-not-permitted/" rel="bookmark">SOLVED: 425 Unable to build data connection: Operation not permitted</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/solved-425-unable-to-build-data-connection-operation-not-permitted/" title="2:37 pm" rel="bookmark"><time class="entry-date" datetime="2011-03-24T14:37:30+00:00">March 24, 2011</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>This error message is specific to the ProFTPd server.  A Chilkat customer found this error within the LastErrorText after trying to upload or download a file, or retrieve a directory listing:</p>
<pre>
425 Unable to build data connection: Operation not permitted
</pre>
<p>The latest ProFTPd server has a configuration setting that by default requires SSL/TLS sessions to be re-used, and this breaks *many* FTP clients, including Chilkat.  The fix to be made on the server is to add the following line to the proftpd.conf and then restart it.</p>
<pre>
TLSOptions NoSessionReuseRequired
</pre>
