	<article id="post-414" class="post-414 post type-post status-publish format-standard hentry category-c-language category-linking tag-cpp tag-linking">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/linking-c-programs-with-the-chilkat-cc-libs/" rel="bookmark">Linking &#8220;C&#8221; Programs with the Chilkat C/C++ Libs</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/linking-c-programs-with-the-chilkat-cc-libs/" title="7:26 am" rel="bookmark"><time class="entry-date" datetime="2012-04-13T07:26:14+00:00">April 13, 2012</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>The Chilkat C/C++ libs are (internally) written in C++.  Therefore, when linking a &#8220;C&#8221; program with the Chilkat libs, the C++ runtime libs must be included.  This is true regardless of the operating system, whether it be Windows, MAC OS X, IOS, Linux, etc.    Different build environments will have different ways of accomplishing the task.  For example, with XCode (on Mac OS X), simply adding an empty source file having a filename with the extension &#8220;.cpp&#8221; is all that is needed.  This blog post will show how to link a &#8220;C&#8221; program on Linux with the Chilkat C/C++ static library.  The solution is to use &#8220;g++&#8221; for the link phase instead of &#8220;gcc&#8221;.  The telltale indicator that the C++ runtime libs are missing is when you see errors involving &#8220;__cxxabiv1&#8221;, such as the following:</p>
<pre>
undefined reference to `vtable for __cxxabiv1::__class_type_info'
</pre>
<p>Here is a simple C program with a simple shell script to build it correctly. </p>
<p><b>buildTest.sh</b> (This is for 64-bit Linux.  Notice that g++ is used for linking, not gcc.)</p>
<pre>
#!/bin/bash -ev

gcc -m64 -Werror -c -fmessage-length=0 -fno-stack-protector -fPIC -o"cTest.o" "cTest.c"

g++ -g0 -m64 -L"/home/chilkat/workspace/chilkat/lib64_static" ./cTest.o -o"cTest"  
    -lchilkat-9.3.0 -lpthread -lresolv
</pre>
<p><b>cTest.c</b></p>
<pre>
#include &lt;stdio.h&gt;
#include "../chilkat/include/C_CkFtp2.h"

void testFtp2Create(void)
    {
    HCkFtp2 ftp;
    BOOL success;

    ftp = CkFtp2_Create();

    //  Any string unlocks the component for the 1st 30-days.
    success = CkFtp2_UnlockComponent(ftp,"Anything for 30-day trial");
    if (success != TRUE) {
        printf("%s\n",CkFtp2_lastErrorText(ftp));
    }

    CkFtp2_Dispose(ftp);
    }


int main(int argc, char *argv[])
{
	testFtp2Create();
	printf("Hello World!\n");
	return 0;
}
</pre>

						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/ios-mac-os-x-link-problems-involving-symbols-such-as-__ztvn10__cxxabiv121__vmi_class_type_infoe/" title="4:12 pm" rel="bookmark"><time class="entry-date" datetime="2012-02-27T16:12:07+00:00">February 27, 2012</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->

				<div class="entry-content">
			<p>If linking produces undefined symbols involving the following:</p>
<ul>
<li>__ZTVN10__cxxabiv121__vmi_class_type_infoE</li>
<li>__Znwm</li>
<li>__ZdlPv</li>
<li>___gxx_personality_sj0</li>
<li>___cxa_begin_catch</li>
<li>__ZTVN10__cxxabiv120__si_class_type_infoE</li>
<li>___cxa_end_catch</li>
<li>__ZdaPv</li>
<li>__ZTVN10__cxxabiv117__class_type_infoE</li>
<li>etc.</li>
</ul>
<p>It means you&#8217;re NOT linking with the C++ runtime libs.  To link with the C++ runtime libs, include an empty source file having a file extension of &#8220;.cpp&#8221;.   For more information, see the <a title="Objective-C Link Notes" href="http://www.chilkatsoft.com/chilkatIos.asp">IOS Objective-C linking notes </a></p>

						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/undefined-symbol-_res_9_query/" title="2:33 pm" rel="bookmark"><time class="entry-date" datetime="2012-01-26T14:33:47+00:00">January 26, 2012</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->

				<div class="entry-content">
			<p><strong>Question:</strong></p>
<p>I had some IMAP code working with Chilkat iOS 9.2.1 version. Tried  upgrading to 9.3.0 (wanted to use the new attachments functionality) and  got the following error:</p>
<p>Undefined symbols for architecture i386:<br />
&#8220;_res_9_query&#8221;, referenced from:<br />
ChilkatResolve::mxLookup(char const*, ScoredStrings&amp;, LogBase&amp;, bool) in libchilkatIos.a(ChilkatResolve.o)<br />
ChilkatResolve::dkimLookup(char const*, StringBuffer&amp;, LogBase&amp;, bool) in libchilkatIos.a(ChilkatResolve.o)<br />
ChilkatResolve::bestMxLookup(char const*, StringBuffer&amp;, LogBase&amp;, bool) in libchilkatIos.a(ChilkatResolve.o)<br />
ld: symbol(s) not found for architecture i386<br />
clang: error: linker command failed with exit code 1 (use -v to see invocation)</p>
<p>It is building fine for Device scheme, and only fails for Simulator scheme.<br />
Any advice?</p>
<p><strong>Answer:</strong></p>
<p>Add &#8220;-lresolv&#8221; to your list of link libraries.</p>
<p>(A quick way to find a solution for unresolved externals such as this is to Google &#8220;Undefined symbol _res_9_query&#8221;.&nbsp;&nbsp;&nbsp;</p>

	<article id="post-386" class="post-386 post type-post status-publish format-standard hentry category-linking category-objective-c category-xcode tag-dylib tag-linking tag-static-libs tag-xcode">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/xcode-linker-will-automatically-choose-dylib-over-static-libs/" rel="bookmark">XCode Linker will automatically choose dylib over static libs</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/xcode-linker-will-automatically-choose-dylib-over-static-libs/" title="2:24 pm" rel="bookmark"><time class="entry-date" datetime="2011-10-24T14:24:02+00:00">October 24, 2011</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>A note about building (linking) a project in XCode:</p>
<p>If a static lib&#8217;s corresponding dylib is in a library search path  during linking, Xcode will link to the dylib instead of the static lib.</p>
<div>The solution is to move the dylib somewhere else.</div>
<div>(from a Chilkat customer) &#8220;I created chilkat-9.2.1-macosx-10.5/lib/dylib and moved the dylibs  into it.  Since my library search path in the project settings is not  recursive, Xcode can&#8217;t find the dylib and links to the static lib.  It&#8217;s  quirky and slightly hackish but you might want to consider structuring  your download the same way to prevent other Xcode users from running  into the problem.&#8221;</div>
<p><div>Note: The next version of the Chilkat Objective-C libs will be structured such that the dylibs and static libs are in separate directories.  Hopefully this will prevent this pitfall (where a user *thinks* the static lib is being linked, but in fact XCode is linking against the dylib).</div>

		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/common-ios-undefined-symbols-error/" rel="bookmark">Common IOS undefined symbols error</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/common-ios-undefined-symbols-error/" title="2:16 pm" rel="bookmark"><time class="entry-date" datetime="2011-10-24T14:16:04+00:00">October 24, 2011</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->

				<div class="entry-content">
			<p><strong>Link Error:</strong></p>
<blockquote><p>Undefined symbols for architecture i386:</p>
<p>&#8220;operator new(unsigned long)&#8221;, referenced from:<br />
-[CkoMailMan init] in libchilkatIos.a(CkoMailMan.o)<br />
&#8220;operator delete(void*)&#8221;, referenced from:<br />
-[CkoMailMan init] in libchilkatIos.a(CkoMailMan.o)</p></blockquote>
<p><strong>Solution:</strong><br />
The Chilkat internals are written in C++. Therefore, your Objective-C application will need to link against the C++ runtime libraries. The easiest way to do it is to include an empty source file with the file extension &#8220;cpp&#8221; in your app target.</p>

	<article id="post-330" class="post-330 post type-post status-publish format-standard hentry category-error-messages category-linking category-vc tag-error-messages tag-linking tag-vc">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/vc-link-problem-fatal-error-c1047-chilkatlib-was-created-with-an-older-compiler-than-other-objects-rebuild-old-objects-and-libraries/" rel="bookmark">VC++ Link problem: fatal error C1047, &#8220;Chilkat*.lib&#8217; was created with an older compiler than other objects; rebuild old objects and libraries&#8221;</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/vc-link-problem-fatal-error-c1047-chilkatlib-was-created-with-an-older-compiler-than-other-objects-rebuild-old-objects-and-libraries/" title="5:05 am" rel="bookmark"><time class="entry-date" datetime="2010-08-25T05:05:47+00:00">August 25, 2010</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>Regarding the following VC++ link problem:</p>
<pre>
“fatal error C1047: The object or library file 'c:\Chilkat\libs\x64\ChilkatRel_x64.lib' 
was created with an older compiler than other objects; rebuild old objects and libraries”
</pre>
<p>This can be resolved by building without the /LTCG flag turned on. Unfortunately, it is not possible to use the /LTCG flag if you are using the Chilkat libraries. Compiling and linking your C++ application without using the /LTCG flag should resolve the problem.</p>

						<h1 class="entry-title"><a href="http://cknotes.com/chilkat-c-libs-link-and-maintain-single-exe/" rel="bookmark">Chilkat C++ Libs &#8211; Link and Maintain Single EXE?</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/chilkat-c-libs-link-and-maintain-single-exe/" title="7:14 am" rel="bookmark"><time class="entry-date" datetime="2009-09-16T07:14:52+00:00">September 16, 2009</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p><strong>Question:</strong></p>
<p>I am developing a single EXE in C++ that doesn&#8217;t make use of any external libraries and including MFC. Can I add your zip libaries to my application and still maintain the application as a single EXE?</p>
<p><strong>Answer:</strong></p>
<p>Yes, the Chilkat C++ libs may be linked directly into your application.  This results in a single EXE containing the Chilkat library code required to satisfy all direct and indirect references.  (The linker includes only the code needed to resolve all references.)  You may run your EXE on any computer because the licensing is royalty-free.  Internally, Chilkat does not use or reference MFC.   (Note: MFC applications may still use the Chilkat libs.  The point is this: if your application does not use MFC, then linking with Chilkat will not cause MFC to be &#8220;pulled-into&#8221; your EXE.)</p>

						<h1 class="entry-title"><a href="http://cknotes.com/choosing-correct-c-library-for-linking/" rel="bookmark">Choosing Correct C++ Library for Linking</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/choosing-correct-c-library-for-linking/" title="4:43 am" rel="bookmark"><time class="entry-date" datetime="2008-05-28T04:43:41+00:00">May 28, 2008</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>When linking with a Chilkat C++ library, you must choose the .lib that matches your project&#8217;s &#8220;code generation&#8221; property setting.  </p>
<p>For example, in VC++ 8.0, these are the Chilkat .lib&#8217;s:</p>
<blockquote><p>
ChilkatDbg.lib<br />
ChilkatDbgDll.lib<br />
ChilkatRel.lib<br />
ChilkatRelDll.lib
</p></blockquote>
<p>If &#8220;Dll&#8221; is in the library name, it means that you are *not* statically linking with the C runtime libs.  Just because your project produces a .dll does not mean you&#8217;ll use the .lib with &#8220;DLL&#8221; in the name.  The choice depends on the &#8220;code generation&#8221; project setting for each build.</p>
<p>I hope you can figure out the difference between &#8220;Dbg&#8221; and &#8220;Rel&#8221; without me explicitly stating it&#8230;</p>
