						<h1 class="entry-title"><a href="http://cknotes.com/chilkat-charsets-character-encodings-supported/" rel="bookmark">Chilkat Charsets (Character Encodings) Supported</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/chilkat-charsets-character-encodings-supported/" title="9:34 am" rel="bookmark"><time class="entry-date" datetime="2015-07-10T09:34:56+00:00">July 10, 2015</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->

				<div class="entry-content">
			<p>In many places in the Chilkat API, there are properties or method arguments for a charset (also known as character encoding).&nbsp; All acceptable and supported charset names are listed here.&nbsp; (The vast majority are likely almost never used.)</p>
<p>Before we list the names and the corresponding code pages, there are a few special values.</p>
<ul>
<li><strong>&#8220;ansi&#8221;</strong> is a string that can be used to indicate the current ANSI code page for the local computer.&nbsp; This might typically be &#8220;iso-8859-1&#8221; or &#8220;windows-1252&#8221; on a US or western European computer, &#8220;shift-JIS&#8221; on a Japanese computer, etc.</li>
<li>In some cases, the special string <strong>&#8220;hex&#8221;</strong> or <strong>&#8220;base64&#8221;</strong> can be used.&nbsp; This indicates that the string contains non-character byte data that is not to be interpreted according to a particular character encoding, but should simply be decoded from the specified binary encoding. (This is a relatively new&nbsp; feature of Chilkat.)</li>
<li>In cases where binary bytes are returned, or when saving to a file, then for any charset having an optional BOM (Byte Order Mark) or preamble, such as &#8220;utf-8&#8221; or &#8220;utf-16&#8221;, the prefixes &#8220;bom-&#8221; or &#8220;no-bom-&#8221; can be prepended to the charset name to indicate whether the preamble/BOM is desired or not.&nbsp; (This is a relatively new&nbsp; feature of Chilkat.)</li>
</ul>

						<h1 class="entry-title"><a href="http://cknotes.com/handling-accented-characters-in-filenames-w-iis-75-ftp/" rel="bookmark">Handling Accented Characters in Filenames w/ IIS 7.5 FTP</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/handling-accented-characters-in-filenames-w-iis-75-ftp/" title="6:14 am" rel="bookmark"><time class="entry-date" datetime="2011-02-20T06:14:54+00:00">February 20, 2011</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>To handle accented characters in filenames correctly, the FTP client must know what character encoding is used to represent these characters (such as utf-8, iso-8859-1, etc.).  The Ftp2.DirListingCharset property indicates the character encoding to be used when interpreting the bytes of a filename (in a directory listing).  By default, it is set to &#8220;ANSI&#8221;, which is another name for  the default multi-byte (non-Unicode) charset of the computer.  When connecting to an FTP server, if the server indicates it supports the &#8220;UTF8&#8221; feature, the DirListingCharset property will be implicitly set to &#8220;utf-8&#8221;.  (You may check the value of the DirListingCharset after connecting to see what it contains.)</p>
<p>There has been some trouble with IIS 7.5 FTP.</p>
<blockquote><p>It turns out that IIS 7.5 FTP has a new UTF 8 advanced option and when I set this to false, all accented characters appears correctly if the DirListingCharset remains at it&#8217;s default value.</p></blockquote>
<p>In this case, it must be that the IIS 7.5 FTP server does not indicate it supports &#8220;UTF8&#8221; feature, and the DirListingCharset remains at its default  value of &#8220;ANSI&#8221; and the FTP server sends ANSI characters.  Since both are in agreement, there is no problem.</p>
<blockquote><p>If this option is set to utf8, whether we set DislistingCharset or not, we don&#8217;t get the accented characters.</p></blockquote>
<p>It is probable that that IIS 7.5 FTP still did not indicate that it supports the &#8220;UTF8&#8221; feature, and after connecting the DirListingCharset is implicitly set to ANSI.  The solution is to force the DirListingCharset to &#8220;utf-8&#8221; by setting this property after connecting.</p>
