	<article id="post-765" class="post-765 post type-post status-publish format-standard hentry category-ios tag-ios tag-linking tag-size tag-static-library">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/ios-cc-static-library-sizes/" rel="bookmark">iOS C/C++ Static Library Sizes</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/ios-cc-static-library-sizes/" title="6:00 am" rel="bookmark"><time class="entry-date" datetime="2016-06-20T06:00:54+00:00">June 20, 2016</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>There is often alarm at the size of the Chilkat static libraries (.a) for iOS. This should not be of too much concern, because after building your app in Release mode, your app&#8217;s executable will NOT grow by the size of the static library. It will only grow by a very small fraction of the total size of the .a</p>
<p>There are some common misunderstandings about static libraries that should be cleared up.</p>
<ol>
<li>You do NOT include the static library (.a) in the package you submit to the app store.</li>
<li>When building your program, Xcode only pulls in the library code that is directly or indirectly used.   (This is true for any C/C++ linker.)   If, for example, Chilkat added functionality for SNMP, Jabber, and Bitcoin, and if this new code caused growth in the .a by 100MB, then it would only cause an increase in your app&#8217;s size if your app actually uses the new code.</li>
<li>You may be unaware of the vast number of already-existing system libs (.a) and the vast sizes if all of these were summed up.  Obviously, these are not included in their entirety in your app&#8217;s executable.  The same applies to the Chilkat static libs.</li>
<li>You can always check to see how much your app&#8217;s executable actually grows in size after linking with Chilkat by examining the size of the executable after linking.</li>
<li>The Chilkat universal lib is composed of both the simulator static libs (x86 + x86_64), and the device libs (arm64, armv7, armv7s).   Your application built for the App Store would not be including alternatives for the simulator.</li>
<li>This Apple Technical Q&amp;A discusses general techniques for reducing the size of an app:  https://developer.apple.com/library/ios/qa/qa1795/_index.html</li>
</ol>

		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/ios-preparing-your-apps-for-ipv6/" rel="bookmark">iOS: Preparing your Apps for IPv6</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/ios-preparing-your-apps-for-ipv6/" title="5:34 am" rel="bookmark"><time class="entry-date" datetime="2016-06-10T05:34:44+00:00">June 10, 2016</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>Chilkat classes that communicate over TCP/TLS include a property named PreferIpv6.   This should be set to YES/true to tell Chilkat to use IPv6 when possible.  (If the underlying DNS resolution provides both IPv4 and IPv6 choices, then Chilkat will by default choose the IPv4 address. To tell Chilkat to choose the IPv6 choice, set the PreferIpv6 property equal to YES/true.)</p>
<p>Note: All Chilkat classes that communicate over TCP/TLS will include a <strong>PreferIpv6</strong> property. This includes Http, MailMan, Imap, Ftp2, Socket, Ssh, SFtp, etc.</p>
<p>For more information, see <a href="http://www.chilkatforum.com/questions/10633/using-ipv6-in-chilkat-apps-for-ios">Using IPv6 in Chilkat Apps for iOS </a></p>

		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/chilkat-v9-4-1-pre-release-for-ios-arm64-is-ready/" rel="bookmark">Chilkat v9.5.0 pre-release for iOS arm64 / x86_64 is Ready</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/chilkat-v9-4-1-pre-release-for-ios-arm64-is-ready/" title="10:32 am" rel="bookmark"><time class="entry-date" datetime="2013-10-09T10:32:50+00:00">October 9, 2013</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>A new build of Chilkat v9.5.0 supporting the arm64 and x86_64 iOS architectures is available:</p>
<p><a href="http://www.chilkatsoft.com/preRelease/chilkat-9.5.0-ios.zip" title="Chilkat iOS supporting arm64">http://www.chilkatsoft.com/preRelease/chilkat-9.5.0-ios.zip</a></p>
<p>The static libraries are provided differently than before.  Chilkat provides static libs for each architecture slice: i386 (for the simulator), armv7, armv7s, arm64, and armv6 (for older versions of iOS).  The libs are located in the directories as shown here:</p>
<pre>
lib/i386/libchilkatIos.a
lib/x86_64/libchilkatIos.a
lib/armv6/libchilkatIos.a
lib/armv7/libchilkatIos.a
lib/armv7s/libchilkatIos.a
lib/arm64/libchilkatIos.a
</pre>
<p>A bash shell script (makeUniversalLib.sh) is  also provided to create a universal static library from the individual slices:</p>
<pre>
#!/bin/bash -ev
cd lib
libtool -static i386/libchilkatIos.a 
    x86_64/libchilkatIos.a 
    armv7s/libchilkatIos.a 
    armv7/libchilkatIos.a 
    arm64/libchilkatIos.a 
    armv6/libchilkatIos.a 
    -o libchilkatIos.a
cd ..
</pre>
<p>You may modify the makeUniversalLib.sh script so that it only includes the slices that are needed.<br />
Your application should then link with the universal libchilkatIos.a that is created by this libtool command.</p>

	<article id="post-415" class="post-415 post type-post status-publish format-standard hentry category-cocoa category-ios category-mac-os-x category-objective-c tag-cocoa tag-ios tag-mac-os-x tag-objective-c">
		<header class="entry-header">
						<h1 class="entry-title"><a href="http://cknotes.com/auto-release-pools-and-background-threads/" rel="bookmark">Auto-release Pools and Background Threads</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/auto-release-pools-and-background-threads/" title="5:49 am" rel="bookmark"><time class="entry-date" datetime="2012-04-18T05:49:09+00:00">April 18, 2012</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>Regarding Objective-C programming for MAC OS X and IOS:</p>
<p>(from https://developer.apple.com/library/mac/#documentation/Cocoa/Conceptual/MemoryMgmt/Articles/mmAutoreleasePools.html )</p>
<blockquote><p>Cocoa always expects there to be an autorelease pool available. If a pool is not available, autoreleased objects do not get released and your application leaks memory. If you send an autorelease message when a pool is not available, Cocoa logs a suitable error message. The AppKit and UIKit frameworks automatically create a pool at the beginning of each event-loop iteration, such as a mouse down event or a tap, and drain it at the end. Therefore you typically do not have to create a pool, or even see the code that is used to create one. There are, however, three occasions when you might use your own autorelease pools:</p>
<p>If you are writing a program that is not based on a UI framework, such as a command-line tool.</p>
<p>If you write a loop that creates many temporary objects.</p>
<p>You may create an autorelease pool inside the loop to dispose of those objects before the next iteration. Using an autorelease pool in the loop helps to reduce the maximum memory footprint of the application.</p>
<p><strong>If you spawn a secondary thread.</strong></p>
<p><strong>You must create your own autorelease pool as soon as the thread begins executing; otherwise, your application will leak objects.</strong> (See “Autorelease Pools and Threads” for details.)</p></blockquote>

						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/ios-creating-a-chilkat-universal-library/" title="7:45 am" rel="bookmark"><time class="entry-date" datetime="2012-02-23T07:45:27+00:00">February 23, 2012</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>The Chilkat IOS library download provides separate static libs for the device and simulator.  The universal lib is not included because (1) it is easy to create from the individual libs, and (2) it reduces the size of the download.</p>
<p>
To create a single IOS Chilkat universal lib, run the following libtool command from the root of the Chilkat directory (where the Chilkat download files were extracted).</p>
<pre>
libtool -static -o libchilkatIos.a libDevice/libchilkatIos.a libSimulator/libchilkatIos.a
</pre>

						<h1 class="entry-title"><a href="http://cknotes.com/iosiphone-setting-ziptempdir-to-the-documents-directory/" rel="bookmark">(IOS/IPhone) Setting Zip.TempDir to the Documents Directory</a></h1>
			
						<div class="entry-meta">
				<span class="sep">Posted on </span><a href="http://cknotes.com/iosiphone-setting-ziptempdir-to-the-documents-directory/" title="7:38 am" rel="bookmark"><time class="entry-date" datetime="2012-02-23T07:38:40+00:00">February 23, 2012</time></a><span class="by-author"> <span class="sep"> by </span> <span class="author vcard"><a class="url fn n" href="http://cknotes.com/author/admin/" title="View all posts by admin" rel="author">admin</a></span></span>			</div><!-- .entry-meta -->
			
					</header><!-- .entry-header -->

				<div class="entry-content">
			<p>The Chilkat Zip methods for writing a .zip will first write the zip to a temp file in the directory specified by the Zip.TempDir property, and then if successful, moved/renamed to the actual output file.  The default value for zip.TempDir is &#8220;.&#8221; (the current working directory) and this is often a directory where it is not possible to create files.  On IOS, a good solution is to set the zip.Temp directory equal to the app&#8217;s Documents directory, as shown in the code snippet below.</p>
<p>
* The reason a temp file is used is because quite often the call to WriteZip / WriteZipAndClose is overwriting an existing .zip archive.  If Chilkat Zip were to write the output file directly, and the zipping failed mid-way, then the existing .zip would be lost.  Using the temp file eliminates this situation.  The existing .zip is overwritten only when the entire new .zip is written.</p>
<p>
* Also, depending on the situation and the sizes of files involved, temp files may be used when unzipping.  The location of the temp directory for both zipping and unzipping is controlled by the TempDir property.</p>
<pre>
- (NSString *)documentsDirectory {
    NSArray *paths = NSSearchPathForDirectoriesInDomains(NSDocumentDirectory,
           NSUserDomainMask, YES);
    NSString *documentsDirectory = [paths objectAtIndex:0];
    return documentsDirectory;
}

...
...
CkoZip *zip = [[[CkoZip alloc] init] autorelease];
...
...
zip.TempDir = [documentsDirectory];

success = [zip WriteZipAndClose];
if (success != YES) {
    [strOutput appendString: zip.LastErrorText];
    [strOutput appendString: @"\n"];
    self.mainTextField.stringValue = strOutput;
    return;
}

...
</pre>
